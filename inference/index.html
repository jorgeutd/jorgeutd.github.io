The KV cache, visualized

Autoregressive decoding is fast because the model never recomputes attention over past tokens — it caches their keys and values. That cache is also the main thing standing between you and a bigger batch, a longer context, or a cheaper GPU. Here is how it grows, and the state-of-the-art tricks that tame it.

Interactive. Everything below runs locally in your browser — no data leaves the page.

1 · Watch the cache fill

The model first prefills the prompt (all tokens in parallel), then decodes one token at a time. Each new token computes attention against every cached key/value, then appends its own — so the cache, and the memory it costs, grow linearly with sequence length.

2 · Size any model’s KV cache

KV cache bytes = 2 × layers × kv_heads × head_dim × dtype_bytes × seq_len × batch. The 2 is for keys and values. Grouped-query attention (GQA) shrinks kv_heads; quantization shrinks dtype_bytes.

3 · State-of-the-art serving

Six techniques that decide throughput, latency, and cost in production. Toggle the interactive ones.

PagedAttention vLLM, Kwon et al. 2023

Reserving one contiguous slab per sequence at max length wastes most of it to internal fragmentation. PagedAttention stores the cache in fixed-size blocks (like OS paging), allocated on demand and shareable across sequences.

Paged: ~4% waste

Continuous batching Orca, Yu et al. 2022

Static batching stalls the whole batch until the slowest sequence finishes, padding the rest. Continuous (iteration-level) batching evicts finished sequences and admits new ones every step, keeping the GPU saturated.

Continuous: GPU busy ~95%

Speculative decoding Leviathan / Chen 2023

A small draft model proposes several tokens; the large target model verifies them in a single parallel pass and keeps the longest correct prefix. Same output distribution, fewer expensive forward passes. EAGLE-2/3 are the dominant production variants in 2026.

idle

MHA → GQA → MQA Shazeer 2019; Ainslie 2023

Query heads can share key/value heads. Fewer KV heads means a proportionally smaller cache — the reason modern models ship with GQA.

GQA: 8 KV heads → 4× smaller cache

Prefix / prompt caching

Requests that share a prefix (a long system prompt, a few-shot preamble, a document) can reuse its already-computed KV instead of prefilling it again — big latency and cost wins for agents and RAG. vLLM ships automatic prefix caching; SGLang generalizes it with RadixAttention, a radix tree over cached prefixes.

Shared prefill computed once, reused N times.

KV cache quantization

Keys and values don’t need FP16. Storing them in FP8 or INT8 (often with per-channel/per-token scales) halves or quarters cache memory, unlocking longer context or larger batches with minimal quality loss.

FP16 → FP8: 2× more context in the same memory.

Chunked prefill & PD disaggregation Sarathi; Splitwise / DistServe / Mooncake

Prefill is compute-bound; decode is memory-bandwidth-bound. Chunked prefill splits long prompts so decode isn’t starved. Prefill/decode disaggregation runs the two phases on separate GPU pools and streams the KV cache between them (NVIDIA Dynamo, Mooncake), reported at 1.5–3× throughput and lower TTFT.

Disaggregated: prefill & decode don’t fight for the GPU

Underneath all of this, FlashAttention (now FlashAttention-3 on Hopper/Blackwell) makes each attention step IO-aware — tiling in on-chip SRAM to cut HBM traffic — speeding up prefill and decode without changing what the cache stores. In 2026, continuous batching, paged KV, and FP8 are table stakes across vLLM, SGLang, and TensorRT-LLM; the differentiators are prefix reuse (RadixAttention), speculative decoding (EAGLE), and KV-cache-centric disaggregation. Together they are why one GPU can serve far more concurrent, longer conversations than the naive math suggests.
