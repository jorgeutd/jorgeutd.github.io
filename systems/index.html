System Design & Architecture

Systems I’ve designed and shipped

Five production AI systems, told the way I’d defend them in a principal-level design review: the real problem, the scale that shaped the architecture, the decisions that mattered (and the ones I rejected), how they fail and recover, how they’re evaluated, and the business outcome. Diagrams are simplified for clarity, not marketing.

  • On-device Sales AI

1 · Offline-first, on-device Sales AI

Grainger · iOS · Apple Silicon · llama.cpp / whisper.cpp

  • 3,000+ sellers
  • on-device only
  • ~$1.2M/yr cloud saved
  • 50k+ hrs/yr

Problem. Field sellers need real-time visit capture and recommendations with no reliable connectivity, on customer sites, without shipping sensitive conversations to the cloud — and cloud STT+LLM was costing ~$1.2M/yr.

 mic ─▶ VAD ─▶ whisper.cpp (streaming ASR) ─▶ transcript buffer
                        │  partial + final          (stable │ unstable)
                        ▼
                 prompt builder ──▶ llama.cpp (GGUF LLM, Metal)
                 (context mgmt)      + GBNF grammar → strict JSON
                        │                        │
                        ▼                        ▼
                 structured visit summary + next-best actions
                        │
                        ▼
                 human review UI ─▶ local store (Core Data) ─▶ sync ▲ (when online)
                                                                 │
                        artifact delivery ◀── checksum + rollback ┘ (model updates)

Key decisions & trade-offs

  • On-device vs cloud: chose local for cost, latency, offline, and privacy; accepted a smaller quantized model (GGUF Q4/Q5) and a benchmarking burden across device tiers.
  • GBNF grammar-constrained decoding: guarantees schema-valid JSON without a fragile parser or retries — the reliability unlock for structured summaries.
  • Swift concurrency isolation: ASR + LLM run off the MainActor on dedicated actors; UI stays responsive; explicit cancellation when a visit ends.
  • Rejected: a always-cloud fallback (defeats the offline goal) and a larger model (thermal/battery/memory budget on iPhone).

Reliability & failure modes

  • Memory/thermal budgeting; unload the model under pressure; cap context window and manage the KV cache.
  • Corrupted/al partial model download → checksum verify + atomic swap + rollback to last-good artifact.
  • App backgrounded / recording cancelled mid-stream → flush stable transcript, discard unstable tail.

Evaluation

  • ASR: WER/CER and real-time factor on held-out field audio; LLM: structured-output validity + summary faithfulness via LLM-as-judge, plus cloud-vs-device parity benchmarks gating each model bump.

Outcome

Eliminated ~$1.2M/yr in cloud inference and saved 50,000+ hrs/yr, with production-grade structured reliability offline.

2 · Enterprise multi-agent email automation

Grainger · LangGraph · Azure OpenAI · FastAPI · Kafka · MCP

  • 48,000+ emails/day
  • ~2.5s response
  • −85% manual
  • same-day fulfillment

Problem. Tens of thousands of inbound order emails/day were processed by hand, blocking same-day fulfillment. Extract intent + attachments, act on order / customer / fulfillment systems (reads and writes), with auditability and no duplicate actions.

 email ─▶ ingress ─▶ Kafka (partition by account) ─▶ consumer group (scales out)
                                                   │
              ┌────────────────────────────────────┘
              ▼
        attachment/doc classifier (EfficientNetV2, 94.9%)
              ▼
        LangGraph router ──▶ specialized agents: [order] [customer] [fulfillment]
              │                          │  tool calls (typed)
              │                          ▼
              │                   MCP tool servers ─▶ enterprise APIs (read/write)
              ▼
        structured output ─▶ confidence gate ──low──▶ human review
              │ high                                        │
              ▼                                             ▼
        result topic ─▶ order lifecycle        retry topic ─▶ DLQ (poison msgs)
              (idempotency keys, distributed tracing throughout)

Key decisions & trade-offs

  • Agents vs deterministic workflow: a router + a few specialized agents, not an open-ended swarm — bounded tool permissions, predictable cost, debuggable traces.
  • “Exactly-once” is a lie; idempotency is real: at-least-once Kafka delivery + idempotency keys on every write tool, so duplicate/redelivered emails never double-book an order.
  • Confidence gate → human-in-the-loop: low-confidence extractions escalate instead of acting; keeps write operations safe.
  • Rejected: synchronous request/response (couldn’t absorb bursts) and one god-agent (unbounded tool access, untestable).

Reliability & failure modes

  • Partition by account for ordering per customer; consumer-group scaling for throughput; backpressure via lag monitoring.
  • Retry topics with capped attempts → DLQ; compensation actions for partially-completed workflows; tool timeouts + circuit breakers on enterprise APIs.

Evaluation

  • DeepEval for tool-selection + tool-argument correctness and task completion; attachment-classifier accuracy tracked with drift alerts; prompt/model versions pinned to eval runs.

Outcome

48,000+ emails/day at ~2.5s, manual handling down ~85%, enabling same-day order fulfillment.

3 · AI release, evaluation & observability control plane

Grainger · S3 · MLflow · Next.js · GitOps / Argo CD

  • weeks → days releases
  • cloud + on-device
  • multi-team

Problem. Multiple AI teams needed one trusted way to decide whether a model, prompt, workflow, or on-device artifact was safe to ship — with governance that didn’t collapse into bureaucracy.

  CONTROL PLANE                                   DATA PLANE
  ┌───────────────────────────────────┐           ┌──────────────────────┐
  │ artifact upload ─▶ S3 (immutable)  │           │ serving (cloud) +    │
  │        │  + checksum               │  promote  │ on-device artifacts  │
  │        ▼                           │ ────────▶ │                      │
  │ registry (metadata DB) ─▶ eval svc │           └──────────────────────┘
  │        │        golden sets, replay,│                    ▲
  │        │        LLM-as-judge, cloud-│  rollback ─────────┘
  │        ▼        vs-device benchmark │
  │ MLflow traces ─▶ approval gate ─▶ GitOps deploy (canary)
  │        └────────▶ dashboards (Next.js) + audit trail
  └───────────────────────────────────┘

Key decisions & trade-offs

  • Control plane vs data plane: the platform decides and records; it never sits in the serving hot path.
  • Immutable, checksum-governed artifacts: promotion moves a verified hash between environments — reproducible, tamper-evident, and safe to roll back.
  • Release gates as thresholds, not vibes: eval scores + human approval gate promotion; the same gate defines the rollback trigger.
  • Avoiding bureaucracy: gates are automated and fast; humans approve only what the evals flag as borderline.

Evaluation & observability

  • Versioned datasets + metrics; replay tests; LLM-as-judge with human-calibrated rubrics; every score is traceable to the exact prompt + model + dataset version (lineage).

Outcome

Cut release cycles from weeks to days and gave product + engineering a shared, auditable basis for what ships.

4 · Insurance virtual assistant (voice + chat)

GEICO · LangGraph · Azure OpenAI · fine-tuned Llama 3.1 · DeBERTa · MCP

  • voice (IVR) + chat
  • 400+ intents
  • read + write ops
  • −14.7% transfers

Problem. Personalized, open-ended support for claims, billing, and policy servicing across IVR and chat — including consequential writes (postpone payment, add a vehicle) — safely and at low latency.

 IVR / chat ─▶ session state (Redis) ─▶ safety in: Prompt Guard + content filter
                        │
                        ▼
        intent classifier (DeBERTa v3, 400+ intents, calibrated)  ──┐
                        │                                                 │ hybrid
                        ▼                                                 ▼
                LangGraph router ─▶ function-calling LLM (fine-tuned Llama 3.1)
                        │                         │ tool calls
                        ▼                         ▼
                read tools ◀──────────── MCP tools (15+) ──▶ write tools
                (claims/policy/billing)     │  server-side authz    │ require confirmation
                        │                    │                       │  + idempotency
                        ▼                    ▼                       ▼
                grounded answer     safety out (filter)      enterprise APIs (txn)
                        └────────── low confidence / risk ─▶ human escalation

Key decisions & trade-offs

  • Hybrid intent routing: a calibrated DeBERTa classifier for the 400+ known intents (cheap, fast, measurable) + LLM routing for the long tail — not one giant prompt.
  • Reads are cheap, writes are consequential: write tools require explicit user confirmation, server-side authorization (never trust model-supplied identity), idempotency keys, and clear transaction boundaries.
  • Safety as layers: Prompt Guard + content filtering on input and output; unsafe/low-confidence paths escalate to a human.
  • Rejected: autonomous writes without confirmation (unacceptable risk) and a pure-LLM router (worse latency, weaker guarantees than a fine-tuned classifier for known intents).

Evaluation

  • Intent F1 (95.7%) with confusion analysis and temporal splits; tool-selection + argument accuracy; task completion; transfer-rate and callback deltas as the business metrics; safety red-teaming for jailbreaks/prompt injection.

Outcome

Live-agent transfers down 14.7% in two months (~$2.7M/yr), callbacks improved 13.6%.

5 · Healthcare RAG & benefits assistant

Quantum Health · AWS Textract · Bedrock · hybrid retrieval · FastAPI

  • 10M+ members
  • −42s handle time
  • +6% calls/agent
  • tenant-isolated

Problem. Call-center agents needed grounded, cited answers from frequently-changing health-plan documents — with strict per-plan permissions and no hallucinated benefits.

 plan docs ─▶ validate + scan ─▶ Textract/Unstructured OCR ─▶ clean/dedupe ─▶ classify
                                                                              │
   chunk + metadata (plan, tenant, effective-date) ─▶ embed ─▶ vector index + BM25
                                                                              │
 agent query ─▶ query rewrite ─▶ hybrid retrieve ─▶ rerank ─▶ context assembly
                                        (permission + tenant filter)          │
                                                                              ▼
                             Bedrock LLM ─▶ grounded answer + citations ─▶ validate ─▶ agent UI
                                                                              │
                                                          eval + monitoring ◀─┘  (freshness / reindex)

Key decisions & trade-offs

  • Hybrid retrieval (dense + BM25) + rerank: plan docs are keyword-heavy (codes, plan names); pure vector search missed exact matches.
  • Permissions at retrieval time: tenant + plan filters are applied before the LLM sees context — isolation can’t depend on the model behaving.
  • Citations + response validation: answers must cite retrieved passages; unsupported claims are blocked to keep hallucination near zero.
  • Freshness: plan documents change; ingestion supports delete + reindex so stale benefits never surface.

Evaluation

  • RAGAS-style metrics — faithfulness, answer relevancy, context precision/recall — plus citation correctness and a curated golden set; retrieval P/R tracked as docs change.

Outcome

Average handle time down 42s and +6% calls handled per agent; a sibling agentic email-triage workflow saved ~$1.5M/yr.
