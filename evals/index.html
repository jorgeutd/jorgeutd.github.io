Evaluation & Governance

How I evaluate production AI systems

Shipping an AI feature is easy; knowing it’s safe to ship — and staying sure as models, prompts, and data drift — is the hard part. This is the field guide I use: what to measure, how to judge it, and how to gate releases on it. The 2026 shift is traceability: every score links back to the exact prompt, model, and dataset version that produced it.

1 · Evaluation ≠ observability ≠ monitoring

These get conflated constantly. They answer different questions and you need all three.

DisciplineAnswersExamples
EvaluationIs the output good against a goal?accuracy on a domain set, RAG faithfulness, hallucination rate on adversarial prompts, safety ratings
ObservabilityWhy did this behavior happen?traces across chains/tools, prompt + context versions, per-step cost/latency, lineage
MonitoringIs it up, fast, and in budget right now?uptime, request/error rates, tail latency, token usage, cost thresholds, alerts

Rule of thumb: monitoring pages you at 3 a.m.; observability tells you what broke; evaluation tells you whether the fix is actually better before it ships.

2 · The offline → online continuum

Quality gates get progressively closer to real traffic. Nothing reaches users without passing the earlier stages.

 golden set ─▶ regression ─▶ shadow ─▶ canary / A-B ─▶ production monitoring
 (curated,       (vs baseline,     (prod traffic,  (small % live,     (live metrics +
  versioned)      block backslide)  no user impact)  guardrails)        human-in-the-loop)
        │                                                                     │
        └──────────────── feedback loop: prod failures become new golden cases ┘
  • offline: fast, cheap, deterministic gates in CI
  • online: catches what offline can’t — drift, real inputs, tail cases

The most valuable dataset you own is the one built from real production failures. Every incident should end with a new golden case so the same regression can never ship twice.

3 · Metrics by system type

There is no single “LLM score.” The metric has to match what the system does — and always pair a quality metric with a business metric.

RAGAS formalized the RAG four — faithfulness, answer relevancy, context precision, context recall (EACL 2024) — and most frameworks now expose them; DeepEval adds G-Eval, task completion, and tool-selection accuracy across single-turn, multi-turn, and agentic cases.

4 · LLM-as-judge — powerful, but calibrate it

Using a strong model to score another model’s output scales evaluation far past human review. It’s the backbone of modern eval — but it is a measurement instrument, and an uncalibrated instrument lies confidently.

  • position bias
  • verbosity/length bias
  • self-preference bias
  • rubric drift
  • cost

How I keep it honest:

  • Report Cohen’s κ, not raw agreement. A 2026 large-scale study of 21 judges found raw exact-match overstates chance-corrected discrimination by 33–41 points (“kappa deflation”); a judge–human κ of ~0.6–0.8 is the acceptable-to-strong band for shipping.
  • Anchor to humans, then scale. Collect human corrections and align rubrics until κ holds (tools like LangSmith Align Evals and MLflow judge alignment make this systematic), then let the judge run.
  • Constrain the task: pairwise comparison and criterion-separated rubric booleans beat free-form 1–10 scores; watch for transitivity violations (A>B, B>C, C>A).
  • Control for bias: randomize option order (position bias persists even in high-consistency judges — the “consistency–bias paradox”), use a cross-family judge (avoid self-preference); verbosity bias is much smaller on 2026 frontier judges than the 2023 literature suggested, but don’t assume it’s gone.
  • Keep humans in the loop for high-stakes/borderline cases; feed adjudications back into the rubric — but beware rubric-induced preference drift from over-editing.

5 · Traceability & release gates

A score you can’t reproduce is a rumor. Treat prompts, datasets, and policies as versioned, first-class assets, and make promotion a gate — not a meeting.

 candidate (model+prompt) ─▶ eval run ─▶ scores vs thresholds
       │                                       │
       │                            pass ──────┴────── fail ─▶ block + report
       ▼                                                        (diff vs baseline)
 register (immutable + checksum) ─▶ approval ─▶ canary ─▶ promote
       │                                                     │
       └──────────── lineage: score ↔ prompt id ↔ model ver ↔ dataset ver
                     rollback trigger = same thresholds crossed in prod

This is exactly the shape of the release/eval control plane I built (immutable S3 artifacts, checksum-governed promotion, MLflow traces, LLM-as-judge, cloud-vs-device benchmarks) — see the Systems page.

7 · The 2026 research frontier

Three shifts are reshaping how serious teams evaluate AI right now:

  • Agentic evaluation — score the trace, not just the answer. Agent-as-a-Judge (Zhuge et al., ICML 2025) uses an agent to give step-by-step feedback over an entire task-solving trajectory and matches human-level reliability on code tasks (DevAI) far more cheaply; Multi-Agent-as-Judge (MAJ-EVAL, ACL 2026) runs debates between persona agents to better align with domain experts and cut single-judge bias.
  • Judge reliability is now a measurement science. A 2026 audit of 21 judges (“Reliability without Validity”) shows raw agreement inflates discrimination by 33–41 points, judge rankings shift by up to 14 places across benchmarks (JudgeBench discriminates ~4.5× more sharply than MT-Bench), and high test–retest reliability can hide severe position bias. The lesson: validate the judge as an instrument (report κ, run a minimum-viable-validation protocol) before you trust its scores.
  • Reliability benchmarks for agents. τ-bench and τ²-bench (dual-control) — plus τ-Knowledge (RAG over unstructured KBs) and τ-Voice — grade tool-agent-user conversations against the final database state and popularized pass^k: the probability of succeeding across k independent trials. Frontier agents still drop sharply as k grows, which is exactly why I gate on consistency, not a lucky single-shot demo.

8 · How I apply this in practice

  • Email automation: DeepEval for tool-selection + argument correctness and task completion, wired into CI as a release gate; classifier accuracy with drift alerts.
  • Insurance assistant: intent F1 + calibration with temporal splits and confusion analysis; safety red-teaming for prompt injection; transfer-rate as the business truth.
  • Healthcare RAG: faithfulness / answer relevancy / context precision-recall + citation correctness on a curated golden set; hallucination held near zero via response validation.
  • On-device Sales AI: WER/CER + structured-output validity + summary faithfulness, with cloud-vs-device parity benchmarks gating every model bump.
  • Across all of them: immutable checksum-governed artifacts, MLflow traces, and lineage so any score reproduces — the difference between “it felt better” and “here’s the diff.”
